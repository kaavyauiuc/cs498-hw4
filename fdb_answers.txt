FDB decouples the Transaction Management System and Storage System to allow each layer to scale or fail independently. The TMS handles transaction coordination by accepting writes, assigning versions, and detecting conflicts, while the SS is responsible for durably storing key-value data and serving reads. This separation means that transaction throughput and storage capacity can be modified separately.

2. The Resolver, which is part of the TMS, is responsible for detecting read-write conflicts between concurrent transactions. This check occurs during the commit process when after a client submits a transaction, the Resolver compares the transaction's read set against writes that have been committed since the transaction's read version, and rejects the transaction if any overlap is found.

3. The read version serves as a consistent snapshot timestamp, i.e. it defines which version of the database the transaction can see, and thus ensures that all reads within the transaction reflect a single point in time. This gives FDB serializable isolation meaning a transaction sees a consistent view of the data as it existed at the read version, and no partial or intermediate states from simultaneous transactions are visible.

4. Snapshot reads do not add keys to the transaction's read set, so they do not participate in conflict detection which means a concurrent write to a snapshot-read key will not cause the transaction to abort. Standard reads however track all read keys and will conflict if those keys are modified concurrently. We choose snapshot reads when we need to observe data without risking false conflicts, such as cases where stale reads are acceptable.

5. FDB recovery is cheap because StorageServers continuously pull and persist mutations from LogServers during normal operations, so by the time a failure occurs, most data is already durable in storage. LogServers act as a short-term write-ahead buffer, and since StorageServers have already applied the log up to a recent point, recovery only needs to replay a very small tail of recent log entries rather than a full redo log, keeping the critical path short, and recovery "cheap".
